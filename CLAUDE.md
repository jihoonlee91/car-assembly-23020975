# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Repository scope

This repo was copied from [mincoding-ai/assemblyCar](https://github.com/mincoding-ai/assemblyCar) as a refactoring exercise. It contains four independent implementations of the same "car assembly" console program: `java/`, `cpp/`, `v2/` (an improved Java version), and `python/`.

**Only `python/` is in scope for changes.** `java/`, `cpp/`, and `v2/` are kept as-is from the original repo for reference (`v2/` in particular shows the pattern this refactor follows) — do not modify them unless explicitly asked.

## Commands (run from `python/`)

```
python main.py                          # run the console app interactively
pip install -r requirements-dev.txt      # install pytest
pytest tests                             # run all tests
pytest tests/test_rules.py -v            # run one test file
pytest tests/test_assemble.py::test_정상_조합이면_PASS  # run a single test
```

Test discovery relies on `python/tests/conftest.py`, which adds `python/` to `sys.path` so test files can do plain `from models import ...` style imports (there's no package/`__init__.py` setup). Run pytest from within `python/`, or reference `python/tests` explicitly from the repo root.

## Architecture (`python/`)

The app is a step-driven console wizard: pick car type → engine → brake → steering → then RUN or Test the resulting combination. The design (see `python/REFACTORING.md` for full rationale) splits this into three layers:

- **`models.py`** — `CarType`, `EngineType`, `BrakeType`, `SteeringType` are enums where each member carries a `(code, label)` pair (`.code` is the menu number, `.label` is the display string). `CarSpec` is a frozen dataclass bundling one selection of each. Adding a new car type/part means adding one enum member — nothing else needs to change, since menus and messages are generated by iterating the enum.
- **`rules.py`** — Compatibility constraints (e.g. "Sedan can't use a Continental brake") are data, not branches: `CompatibilityRule(predicate, message)` objects in the `RULES` list, checked by `CompatibilityChecker.check(spec) -> list[str]`. New constraints are added to `RULES`, not by editing `if` chains.
- **`assemble.py`** — `Assemble` is an abstract base class holding all step/menu flow logic (`Step` enum for the wizard's 5 states, `STEP_PART_ENUM`/`STEP_HEADER`/`STEP_ERROR_PREFIX`/`SELECTION_MESSAGE` dicts driving per-step behavior generically). It has no I/O of its own — `print_`, `println`, `read_line`, `clear`, `delay` are abstract methods implemented by subclasses:
  - **`assemble_console.py`** (`AssembleConsole`) — real stdin/stdout/`time.sleep`. `main.py` is the only entry point, and just does `AssembleConsole().run()`.
  - **`assemble_testable.py`** (`AssembleTestable`) — feeds a list of scripted inputs and captures all output into one string, returned from `.run(inputs)`. This is what the test suite drives; it never uses subprocess or real stdin.

When changing behavior, prefer extending `RULES`/enum members over adding new branches in `assemble.py` — that's the whole point of this layering.

### Known intentional deviations from the original `assemble.py`

Documented in `python/REFACTORING.md` under "원본 대비 알려진 차이": brake/steering selection-confirmation messages were normalized to title case (originally inconsistent — uppercase in the confirmation message, title case in the RUN summary). Everything else (error message wording incl. 은/는 particles, PASS/FAIL text format, exact menu ordering) is intended to match the original byte-for-byte — treat any other mismatch as a bug, not a deliberate change.
